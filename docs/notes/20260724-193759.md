## Balancing Feature Engineering, Velocity Scoring, and Drift in Fraud Detection

In fraud detection, balancing static features, velocity scoring, and model drift is tricky. Static features (e.g., user age, account creation date) are foundational but often lack real-time context. Velocity scoring addresses this by tracking actions over time, like flagging a user making 50 transactions in 10 minutes. However, implementing velocity scoring introduces complexity around time-window definitions and data granularity. For example, too short a window can flag genuine bursts of activity (e.g., flash sales), while too long a window dilutes fraud signals.

Drift is another challenge. Fraud patterns evolve rapidly, especially in response to detection strategies. I've seen velocity thresholds work well initially, then degrade after fraudsters adapt. A practical fix I've used is logging historical velocity patterns to periodically retrain thresholds based on actual fraud cases.

One gotcha: feature drift can break assumptions. If a velocity feature like "logins per minute" suddenly spikes across all users (e.g., after a UI change), it can falsely inflate fraud risk. Monitoring these shifts and having a retraining pipeline is critical to keeping both precision and recall in check.
